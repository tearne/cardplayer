# MP3 buzz

## Intent

Some MP3 files play with a loud high-pitched buzz instead of music, while others on the same card play cleanly. The first observed example reports as MPEG-1 Layer 3, 44.1 kHz, 64 kb/s CBR, **mono** per `mediainfo`. The aim is to identify what's different about these files and make them play correctly.

## Approach

### Branch on `channels` in `ConsumeSample`

When the decoder reports mono (`channels == 1`), take `sample[0]` directly; for stereo, keep the existing `(L+R)/2`. Reason: mono decoders are not required to fill `sample[1]`, so averaging mixes a real sample with whatever happens to be in `sample[1]` — the buzz on the 64 kb/s mono file matches this.

### One-off format logging in `startPlayback`

After `gen->begin()`, print rate / bits / channels (from the `AudioOutput` base) once per track. The line stays in the codebase rather than being stripped at change close — the cost is one `Serial.printf` per track start, and the next unusual file will be much faster to triage. Reason: the firmware currently has no window into what the decoder thinks each file is.

## Plan

- [x] Update `ConsumeSample` in `audio_output_m5.h` to branch on the `channels` member: take `sample[0]` directly when mono, average `(sample[0]+sample[1])/2` when stereo.

- [x] Add a one-line format log in `startPlayback` (`main.cpp`) after `gen->begin()` succeeds, printing `g_out`'s rate, bits, and channels.

## Log

- First on-device test plays cleanly, but the format line reports `ch=2` rather than the expected `ch=1` — so the new mono branch in `ConsumeSample` did not fire for this file. Either the file under test isn't the original buzzy one, or the helix MP3 decoder upmixes mono → stereo internally (in which case the fix is dormant for current files but still correct defensively). Awaiting retest of the originally-buzzy file to settle this.

- Confirmed it is the same file: previously buzzed, now plays cleanly, but still reports `ch=2`. Cause-effect unclear — the fix shouldn't have helped a `ch=2` path. Suspect either the original buzz was intermittent, or there was an unrelated incidental fix in the rebuild. Need power-cycle + repeated-play validation to establish whether the fix is real or coincidental.

- Validation: across multiple files, power cycle, and repeated plays, no buzz reproduces.

- Root cause of the misleading log identified: `AudioGeneratorMP3::begin()` unconditionally calls `output->SetChannels(2)`, then the actual file's channel count is set later (line 182 of `AudioGeneratorMP3.cpp`) when the first frame is decoded. Our log fired right after `begin()` returned, capturing the placeholder rather than the real value. So the file *was* mono, our mono branch *did* fire, the fix *was* the fix — we just couldn't see it.

- Format log relocated to fire on the first `ConsumeSample` after each track change. New `resetFormatLog()` method on `AudioOutputM5CardputerSpeaker` is called from `startPlayback` after `gen->begin()`; a `_log_pending` flag is checked at the top of `ConsumeSample` and prints once per track. Public `rate()` / `bits()` / `numChannels()` getters dropped — the printing moved inside the class so external accessors are no longer needed.

- `ConsumeSample`-based log still reported `ch=2`. Cause: `AudioGeneratorMP3::loop()` calls `output->ConsumeSample(lastSample)` at the very top — the *previous* frame's leftover sample — before decoding any new frame. `GetOneSample` (which calls `SetChannels(actual)`) only runs afterwards. So the first `ConsumeSample` still fires with `channels` at the placeholder value.

- Hook moved from `ConsumeSample` to a `SetChannels` override. The decoder calls `SetChannels(2)` from its `begin()` (placeholder) and then `SetChannels(actual)` from `GetOneSample` after the first frame decodes. Since `resetFormatLog()` runs between those two calls, only the second (real) call sees `_log_pending = true` and logs. `SetRate` is called immediately before `SetChannels` in `GetOneSample`, so `hertz` is also current at log time.

## Conclusion

The originally-buzzy file now reports `ch=1` via the diagnostic line, confirming the mono branch in `ConsumeSample` is what's keeping it clean. Buzz no longer reproduces on any tested file.

Diagnostic plumbing took two relocations to land in the right place — see the Log. The final design: `resetFormatLog()` arms the print from `startPlayback`, and a `SetChannels` override fires it on the first decoder-driven format update for each track.
