# Canvas at 8 bpp

**Mode:** Formal

## Intent

Reduce the off-screen canvas from 16 bpp (RGB565) to 8 bpp (palette-indexed), reclaiming ~32 KB of internal RAM. The whole UI uses only ~12 distinct colours, so 256-entry palette is comfortably oversized for the actual range we draw with. Pushes to the panel still emit RGB565 тАФ the conversion happens at push time via the LovyanGFX palette path.

This is the second of two memory-reclamation changes prompted by the FLAC silent-skip investigation. The first (lazy-load fuzzy index) is a separate change; this one focuses on the canvas. They're independent and either can land alone, but together they roughly double the heap headroom available for decoder allocations.

## Approach

### `rgb332_1Byte` over `palette_8bit`

LGFX offers two 8 bpp modes:

- `rgb332_1Byte` тАФ R3 G3 B2 packed in one byte. 256 fixed colours, no palette table. Drawing functions accept RGB565 and quantise automatically.
- `palette_8bit` тАФ 8 bpp index into a 256-entry palette of RGB565. Exact colours but drawing functions take indices, so every colour constant in the codebase would change.

We pick `rgb332_1Byte`. Far less code churn тАФ existing RGB565 colour constants and call sites work unchanged. Cost: some colour drift on subtle tones (slate-blue frames, dim greys). The user has signed off on accepting visual change for memory.

### Push-to-display path

- `presentFrame()` uses `pushSprite(&display, 0, 0)` тАФ already handles cross-depth conversion transparently. No change needed.
- `presentRows()` currently bypasses that path: it reads the canvas buffer as `uint16_t*` and calls `pushImage` directly, assuming RGB565 layout. That breaks at 8 bpp. Replaced with a `setClipRect` + `pushSprite` pattern so depth conversion happens automatically; LGFX writes only the pixels inside the clip.

### Expected costs

- ~32 KB recovered in internal RAM тАФ the headline win.
- Conversion CPU per full canvas push: ~2тАУ3 ms (one byte тЖТ one RGB565 word, 32 K pixels). `presentFrame` does this once per call; `presentRows` pays the full canvas conversion cost too because it now goes via `pushSprite + clip` (clipping skips *writing* but the source iteration still happens).
- Marquee and waveform redraws will get marginally more expensive тАФ modest single-digit % CPU on the UI core for the affected windows. Verify on device.

### Fallback if visual quality is poor

If the slate-blue/slate-green frames or the dim greys look bad after quantisation, we switch to `palette_8bit` and absorb the per-colour-constant change. Not in this change's scope; flagged as a follow-up if needed.

## Conclusion

Canvas switched from 16 bpp (RGB565) to 8 bpp (RGB332) тАФ `g_canvas.bufferLength()` confirms 32 400 bytes, halved from ~65 000. Visuals look fine per user, no follow-up to switch to palette mode needed. `presentRows` now uses `setClipRect` + `pushSprite` instead of the raw-buffer `pushImage` path that assumed RGB565 layout.
